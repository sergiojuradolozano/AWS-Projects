Static Web Deployment with S3 and CloudFront

Serverless Architec​ture

For this website to load fast, Route 53 routes traffic to the region with the lowest possible latency for the user. Then, CloudFront is used for fast content delivery. The website repository is stored in a private bucket that is routed to the CloudFront distribution. This route allows users to see the content without having access to the bucket containing the website repository.

The importance of Content Delivery Network

A content delivery network locates clustered servers edge locations) regional or globally. The servers are used to store webpage objects temporarily, known as cache.  AWS provides CloudFront as its CDN. As it can be obseerved in the image, the user is geographicaly far from the bucket, nonetheless with the use of edge locations the user will be connected to the closer edge location. When the user sends the request, if the content (object) is in the cache, the edge location it returns it to the uders, and if not it forwardes the request to the origin server, which wil return it to the edge location and the edge location to the user. At the point in time I am writing this page, AWS has 400+ edge locations.

1

Create SSL Certificate

The domain name that I transferred to Route 53 is in a hosted zone. The identity of this site at this point is not authenticated. For this website to be protected, it requires an SSL certificate. I used AWS Certificate Manager for this project to request a public SSL certificate. When submitting the request, I added the domain with and without the “www” alias and used DNS validation and RSA 2048. After the request was submitted, the CNAME records created on that request needed to be added to Route 53 for the SSL certificate to take effect on the domain. It is essential to mention that although the primary users of this site will be located in the UK, the certificate request needs to be made in the North Virginia Region, otherwise the certificate will not appear in the dropdown menu under Custom SSL certificate on CloudFront (see CloudFront later on).

2

Create a Private Bucket

Looking at static website architectures using S3, I have noticed that some of them have their repository in public buckets. I would not consider this a good practice for security. Users do not need access to the bucket to view website content following the zero trust and least privilege principles. By using the CloudFront endpoint, users will be able to access content indirectly, as we will see in the next step. As a result, the website would be less exposed to cyberattacks like ransomware.This table shows the main values when setting the S3 bucket. 
The selected region was London, as it is closest to the expected audience of this website. I gave the name of the bucket my full name and included name tags for monitoring purposes. Bucket Versioning was disabled as this site does not require version control given the minimal changes in content. The bucket is blocked from public access and server-side encryption has been disabled, since enabling SSE-S3 or SSE-KMS might incur charges. Note that if this site contains sensitive data, enabling server-side encryption would be a wise idea.

3

Set up a CloudFront Distribution

After requesting the certificate and creating the S3 bucket, I set up the CloudFront distribution to restrict access to CloudFront only. This table shows the main values selected when configuring the distribution. As can be seen, the origin domain is the bucket. At this point, access control needs to be restricted, so only CloudFront can access it. That is the reason why the bucket does not need to be public, and the web hosting option within the bucket settings should be disabled. Note that I created a new control setting as I did not have Origin access control. The newly created policy by AWS needs to be added to the bucket policy (See step 5). Finalising the configuration of CloudFront distribution, I enabled the Origin Shield and selected the region of London. This will protect the availability of the site by reducing duplicate requests for the same content.  

4

S3 Policy

The Launch Template script automates the Auto-Scaling Group functionality. If the running instance fails, the Auto-Scaling group will deploy a new instance from the same launch template to maintain this site highly available. For that reason, the instance needs to be automated to ensure that the LAMP Server and WordPress will be automatically installed in the newly created instance. In the last section of the Launch Template creation, under "Advanced Details", the script that can be seen when clicking on "View Automation Script" is pasted into User Data. Without placing the script in the User Data box, the EC2 instance that the Auto-Scaling group would deploy would not have a LAMP server installed, nor WordPress, and that would have to be manually added, which is time-consuming and not responsive.

7

Configuring the Application Load Balancer

Before the creation of the Load Balancer, a Target Group needs to be created in the same VPC. The selected Target Type for the target group is "Instance", and the rest of the configuration is set to default. The performance of this infrastructure would be improved by an Application Load Balancer. In this case, the Load Balancer is set to face the internet with the EC2-Security Group as a security measure. Using Port 80 (HTTP), traffic will be forwarded to the target group that was previously created. When the load balancer is active, it is added to the Auto-Scaling Group. 8

Transferring my registered domain to AWS with Route 53

When designing the architecture for this project, I had a registered domain. Therefore, I decided to migrate the DNS service to Amazon 53. This is because it would help users from other regions access this site with low latency, given that AWS has a global network of DNS servers located across the world. To achieve that, I created a hosted zone with my Domain Name and added the endpoint of the Application Load Balancer as the A record for "www.sergiojuradolozano" and "sergiojuradolozano.com" to redirect traffic to the Load Balancer that was configured in the previous step.
