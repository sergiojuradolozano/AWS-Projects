Static Web Deployment with S3 and CloudFront

Serverless Architec​ture

For this website to load fast, Route 53 routes traffic to the region with the lowest possible latency for the user. Then, CloudFront is used for fast content delivery. The website repository is stored in a private bucket that is routed to the CloudFront distribution. This route allows users to see the content without having access to the bucket containing the website repository.

The importance of Content Delivery Network

A content delivery network locates clustered servers in regional or global locations. These servers are used to temporarily store data, known as "cache". AWS provides CloudFront as its CDN resource. As can be observed in the image, the user is geographically far from the bucket. However, with CloudFront, the user will be connected to the closest edge location. If the user (requester) sends the request and the content is stored locally, it will return it to the user. When the requested content is not present at that location, it forwards the request to the origin server. This will return it back to the edge location, which will send it back to the user. At the time I am writing this page, AWS has 400+ edge locations.

Configure SSL Certificate

The domain name that I transferred to Route 53 is in a hosted zone. The identity of this site at this point is not authenticated. For this website to be protected, it requires an SSL certificate. I used AWS Certificate Manager for this project to request a public SSL certificate. When submitting the request, I added the domain with and without the “www” alias and used DNS validation and RSA 2048. After the request was submitted, the CNAME records created on that request needed to be added to Route 53 for the SSL certificate to take effect on the domain. It is essential to mention that although the primary users of this site will be located in the UK, the certificate request needs to be made in the North Virginia Region, otherwise the certificate will not appear in the dropdown menu under Custom SSL certificate on CloudFront (see step 3).

2

Create a Private Bucket

Another benefit of using CloudFront with S3 is that the S3 bucket can be private. It could be considered that setting up a public bucket for a static website is not a recommended course of action. Users do not need access to the bucket to view website content following the zero trust and least privilege principles. By using the CloudFront endpoint, they will be able to access content indirectly, as we will see in the next step. As a result, the website would be less exposed to cyberattacks like ransomware.

This table shows the main values when setting the S3 bucket. The selected region was London. I named the bucket "sergiojuradolozano" and included name tags for monitoring purposes. Bucket Versioning was enabled as I will use version control when a continuous delivery pipeline is automated with CodePipeline. Public access to the bucket has been blocked, and server-side encryption has been disabled since no data needs to be protected at rest and in transit. Note that if this web application contains sensitive data, enabling server-side encryption would be a wise idea.

3

Set up a CloudFront Distribution

After requesting the SSL certificate, creating the S3 bucket and setting up a continuous deployment pipeline, I created a CloudFront distribution to restrict access to CloudFront only and increase cache performance with edge locations across the globe. This table shows the main values selected when configuring the distribution. As can be seen, the selected origin domain is the S3 bucket. At this point, access control needs to be restricted, so only CloudFront can access it. That is the reason why the bucket does not need to be public, and the "Web Hosting" option within the bucket settings should be disabled. 

Note that I created a new control setting as I did not have Origin access control. The newly created policy by AWS needs to be added to the bucket policy (See step 4). Finalising the configuration of CloudFront distribution, I enabled the Origin Shield and selected the region of London. This will protect the availability of the site by reducing duplicate requests for the same content. 

Another important apect when configuring the distribution is to create an invalidation to update the distribution when object are updated. To do that, I opened the distribution and under "Invalidations" tab, I created an invalidation object path pointing at all the object in the bucket with "/*".

Given that AWS automatically generates the policy after the creation of a new origin access control, I copied the policy and paste it into the bucket policy.

5

Create a repository with CodeCommit

The benefits of using version control to update any changes  made on the website repository inclined me to set a continuous deployment pipeline with CodePipeline to automate the S3 bucket everytime I commit and push any change. At this point, the repository can be empty as the pipeline needs to be created. When the continuous development pipeline is created we can always come back to upload the objects of the bucket.

To watch a demonstration of how to migrate Git to CodeCommit, please, go to AWS Hands-On Projects. 

6

Set a Continuous Deployment Pipeline with AWS CodePipeline

The benefits of using version control to update any changes made on the website repository encouraged me to set a continuous deployment pipeline with CodePipeline to automate the S3 bucket every time I commit and push any change on the website repository. To watch a demonstration of how to migrate Git to CodeCommit, please go to AWS Hands-On Projects. 

The pipeline was configured to receive the input artifact from CodeCommit, pointing at the created repository and the right branch. Cloudwatch was the resource selected to detect changes in the source code. The output artefact format was set to default as it does not require including metadata about the repository. I skipped the build provider option as this web application is serverless.

For the changes to take place, the S3 bucket needs to be selected as the Deploy provider, followed by the corresponding values of the S3 that were created previously. In this case, I selected the option to "extract file before deploy" to ensure that the objects are directly uploaded into the bucket. The additional configuration when adding the deploy stage is left blank as CloudFront was set to control traffic, alongside DNS for performance.

Alternative: Optimise the solution

There could be case scenarios where by regulatory compliance, data should be served from the same country, or the traffic needs to be decoupled. In such cases, ​this architecture would would be a great solution. 

As it can be observed in the image, users from USA, UK or Germany would be respectively redirected to the bukets that are in their respected regions. Lambda@edge function could be triggered with CloudFront to decouple the content delivery network. Therefore, users will be redirected to the bucket that is in their regions or to the origin if they are outside of any of the Buckets.
