How this site was deployed with AWS EC2

Network Architecture

Based on 4 Public Subnets from different Availability Zones (AZs) that are routed to the internet gateway.

App Architecture

EC2 and RDS instances are running on the back end. The EC2 instance uses a load balancer and an autoscaling group to create another instance if the current one stops. RDS is set to back up the database every 7 days to make this WordPress site highly redundant.

Based on the Free Tier

This project was limited to the Free Tier capabilities provided by AWS. The only cost incurred was for the "hosted zone" on Route53, which is $0.50/month. This site was hosted on one EC2 instance and a single availability zone RDS instance to comply with the Free Tier requirements. The security, redundancy, and availability of this site could have be improved by placing the EC2 and RDS instances on different private subnets while using a bastion server on an instance that is in a public subnet with access to internet. However, that would involve the cost of using a NAT gateway. The RDS instance, on the other hand, could be replicated in different availability zones across the VPC. 
The processors and memory of EC2 and RDS are set close to the Free Tier boundaries to optimise their performance at a 0 cost (see tables in steps 3 and 4).

1

Setting VPC,subnets, Internet Gateway and Route table

I created the VPC in the region of London, as it was the closest to my 'potential viewers' (users within the UK) to ensure low latency. However, users from other regions would experience low latency too, given that the Application Load Balancer was linked to my registered domain name via Route 53, which makes the site highly available globally (see step 8). When configuring the VPC, the CIDR is set to the standard "/16" notation, although this could have been set to a reduced amount of IP addresses as the number of resources within the VPC was limited. 
As it can be observed in the table, I set up 4 public subnets in different zones with a CIDR of /28. Each public subnet was assigned to the main route table which points to the internet gateway. Please note that I edited each public subnet setting to enable auto-assignment of IPv4 addresses and explicitly mapped them to the route table that was connected to the internet gateway.

2

Creating Security Groups

The website had a two-tier architecture. The inbound rules of the web app (EC2 instance) were set to allow inbound traffic from ports 22, 80 and 443, corresponding respectively to SSH, HTTP and HTTPS. While HTTP and HTTPS allowed traffic from the Internet, the SSH port was only accessible from my IP for security purposes. As with the EC2, the security group of the RDS instance was set up to allow inbound traffic from port 3306 (MYSQL/Aurora) only from the security group of the EC2 instance that will be deployed in the following steps to add that extra layer of security.

3

RDS Creation

At this point, I faced the following problem: creating replicas of the main RDS instance is outside the Free Tier scope. After risk assessing the situation, I decided to accept the risk of having a single RDS point of failure. This is because this website does not hold confidential data such as credit card information or personal data. As can be observed, the table represents the main steps when configuring the creation of an RDS instance. This instance was created in the public subnet 4. The subnets assigned to the RDS instance correspond to the 4 subnets mentioned previously. The main reasons why RDS was hosting WordPress at the backend were the automated backups and patching provided by AWS. Backups were set to be retained for 7 days, and assigned 20 GiB of magnetic storage which are included in the Free Tier. The selected existing security group was "RDS-Security-Group", which was created in the previous step.

4

Launch Template Creation

The EC2 instance was designed to be highly available and redundant within the Free-Tier architecture. The creation of a launch template was used in the auto-scaling group to keep 1 instance running all the time (see step 6). To automate the LAMP server and WordPress installation and its configuration to the RDS instance, I ran a bash script to add it to 'User Data' in the configuration of the Launch Template (see step 5). For the creation of the Launch Template, the selected AMI was AWS Linux 2, with t2.micro as the instance type. 
I created a dedicated key pair and placed it in a safe place to SSH into my instance when needed. Then, I selected the VPC that was created previously and the availability zone was not defined as the instance will be set to launch automatically in either Public Subnet 1, 2 or 3 with the auto-scaling group. Lastly, the selected security group is EC2-Security-Group which was set up in step 2 to allow traffic from ports 22, 80 and 443.

5

Automating the backend installation and configuration (User Data)

The Launch Template script automated the Auto-Scaling Group functionality. If the running instance fails, the Auto-Scaling group will deploy a new instance from the same launch template to maintain this site highly available. For that reason, the instance needed to be automated to ensure that the LAMP Server and WordPress will be automatically installed in the newly created instance with minimal manual configuration. In the last section of the Launch Template creation (under "Advanced Details"), I pasted the script into "User Data". Click here to view the script. Without placing the script in the User Data box, the EC2 instance would not be deployed with LAMP server installed, nor WordPress, and that would have to be manually added, which is time-consuming and not scalable.

6

Configuring the Auto-Scaling group

The EC2 instance running at any given time would be replaced with another EC2 (with the same configurations at the backend) if the current instance does not pass the EC2 and Elastic Load Balancer (ELB) checks. To configure the Auto-Scaling Group, all the Desired, Minimum and Maximum capacities will be set to 1 as can be observed in the table. Secondly, the selected template will be the one from step 5. For the network, the chosen availability zones and subnets correspond to the ones created in step 1. The load balancer will be configured separately in the following step and then assigned to the Auto-Scaling group.

7

Configuring the Application Load Balancer

Before the creation of the Load Balancer, a Target Group needs to be created in the same VPC. The selected Target Type for the target group is "Instance", and the rest of the configuration is set to default. The performance of this infrastructure would be improved by an Application Load Balancer. In this case, the Load Balancer is set to face the internet with the EC2-Security Group as a security measure. Using Port 80 (HTTP), traffic will be forwarded to the target group that was previously created. When the load balancer is active, it is added to the Auto-Scaling Group. This graph visually represents their functionality.

8

Transferring my registered domain to AWS with Route 53

When designing the architecture for this project, I had a registered domain. Therefore, I decided to migrate the DNS service to Amazon 53. This is because it would help users from other regions access this site with low latency, given that AWS has a global network of DNS servers located across the world. To achieve that, I created a hosted zone with my Domain Name and added the endpoint of the Application Load Balancer as the A record for "www.sergiojuradolozano" and "sergiojuradolozano.com" to redirect traffic to the Load Balancer that was configured in the previous step. Shifting to S3

Please notice that the automation could be optimised by adding the installation of an SSL certificate since it needs to be installed on the running server. Additionally, WordPress could be replaced by the website repository, which would add to the ease of working with version control. The automation for this site would have been done in a few lines of code since PHP and a database would not be required. With this alternative, cloning the repository after installing Apache and Certbot could be clearly more practical. 

The simplicity and practicality of using S3 to host a static website encouraged me to shift from RDS to S3. Therefore, after the deployment of this site with RDS, I deleted the resources and used the web repository to host it on an S3 bucket. By the time you are reading this, this website is on S3.
